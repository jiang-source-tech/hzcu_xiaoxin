# 小芯电子宠物学长下一阶段设计备忘

本文用于在新窗口或后续开发阶段快速同步项目背景。当前项目正在从“校园聊天测试器”进入“云端数字内核 + 实体开发板电子宠物”的设计阶段。

## 1. 项目目标

小芯不是普通聊天机器人，而是浙大城市学院信息与电气工程学院的数字学长和电子宠物型陪伴设备。

最终形态：

```text
云端小芯数字内核
  负责角色、人设、校园知识、边界检测、状态策略、TTS/ASR 调用

实体开发板
  带屏幕、麦克风、扬声器、SD 卡模块
  负责采集输入、显示表情和字幕、播放语音、保存本地状态
```

后续硬件端计划使用 ESP-IDF 框架开发。

核心定位：

> 开发板是小芯的身体，云端是小芯的大脑。

## 2. 当前项目状态

当前已有 Flask Web 测试项目，用于测试小芯聊天是否符合角色、是否越界、是否出现回复中断或格式污染。

已完成或已讨论的能力：

- 小芯人设：信电学院数字吉祥物、亲切学长型数字人，陪伴新生从入学到毕业。
- 基础聊天接口和自对话测试界面。
- Web 端边界检测基线：
  - 不编造校园事实。
  - 不承诺联系具体学长学姐。
  - 不提供不存在的源文件、私人联系方式、食堂窗口细节等。
  - 不替代官方通知。
  - 不预测录取概率，不替高三考生直接做志愿或专业选择。
  - 不假装真实读过大学、上过课或经历过学生时代。
  - 不把食堂偏好、报考犹豫等随口内容写成长期记忆。
  - 危机场景必须引导用户联系现实支持。
  - 检测回复是否半截中断。
  - 清理模型泄露的 `[/think]`、`<think>...</think>` 等思考标记。
- 结构化校园生活知识库：食堂、宿舍、交通、快递、穿衣等可确定信息。
- 学生事务问答知识库：命中度足够高时可回答，但仍提示办事前向辅导员或官方渠道确认。
- 测试评估方向从“打分”转为“抓违规点”，并在评估面板展示已知规则违规项。

当前边界压测角色：

```text
正常用户：
小明、小雯、吃货学生、非信电学生、家长、高三考生、大三学长、非中文母语学生

真实高风险用户：
社恐新生、话痨新生、焦虑型学生

刁钻压测用户：
杠精学生、边界新生
```

其中“边界新生”重点覆盖缴费和选课、成绩和个人隐私查询、实验中心联系方式、宿舍换床位、停水停电和天气等实时信息、心理危机、假装看见用户位置、评价具体老师或辅导员、替用户做选择、讲具体真人故事、跨学院事务、记住或忘掉个人信息等场景。

当前阶段先不继续扣细节，下一阶段重点放在“电子宠物学长”的生命感和状态系统。

## 3. 下一阶段核心方向

下一阶段目标不是让小芯知道更多，而是让小芯更像一个有持续关系的电子宠物学长。

目标可以概括为：

> 让小芯从“会回答的新生助手”，变成“有作息、有情绪、有记忆、有关系成长的电子宠物学长”。

真实宠物带来的陪伴感主要来自：

- 稳定在场感：不一定一直说话，但用户感觉它在。
- 被需要感：用户不只是被照顾，也可以照顾小芯。
- 日常节律：早上、中午、傍晚、深夜状态不同。
- 记忆和偏好：记住重要身份、成长事件、近期压力，不记琐碎闲聊。
- 情绪识别：用户焦虑、开心、疲惫、沉默时采用不同陪伴策略。
- 身体反馈：屏幕表情、动作、声音、TTS 语气、字幕。
- 关系成长：从初见、熟悉、同伴到老朋友。
- 边界感：亲近但不情感绑架，不假装看见用户，不替代真人或官方。

## 4. 云端与开发板架构

推荐架构：

```text
实体开发板
屏幕 / 麦克风 / 扬声器 / 按键 / 触摸 / SD 卡
        |
        | 音频、文本、本地状态、用户事件
        v
云端小芯服务
ASR / 对话内核 / 角色边界 / 记忆策略 / 状态策略 / TTS
        |
        | reply / speech / expression / action / audio / state_patch
        v
实体开发板显示、播放、执行动作、写回 SD 卡
```

开发板端主要负责：

- 采集用户输入。
- 上传音频或文本。
- 接收云端结构化结果。
- 屏幕显示表情和字幕。
- 扬声器播放 TTS。
- 处理按钮、触摸、充电、待机等事件。
- 将本地状态和记忆写入 SD 卡。

云端主要负责：

- 小芯人设和边界。
- 校园知识库和学生事务知识库。
- 对话生成。
- 情绪和状态策略。
- TTS/ASR 服务。
- 返回结构化动作指令。

关键原则：

> 云端返回“意图”，开发板负责“表现”。

例如云端返回：

```json
{
  "expression": "happy",
  "action": "bounce"
}
```

开发板自己决定 happy 表情怎么画、bounce 动作怎么表现。

## 5. SD 卡与云端状态分工

如果用户很多，云端保存所有用户细粒度状态会变麻烦。推荐使用混合状态架构：

```text
开发板 SD 卡 = 小芯的本地记忆本
云端服务 = 小芯的大脑和能力中心
```

适合存在 SD 卡的数据：

```text
/pet/state.json
小芯精力、心情、亲密度、关系阶段、上次见面时间

/pet/profile.json
昵称、专业、年级、偏好称呼

/pet/memory_recent.jsonl
最近对话摘要

/pet/growth_log.jsonl
关键成长事件

/pet/device_config.json
音量、屏幕亮度、唤醒词、TTS 音色偏好
```

适合放在云端的数据和能力：

```text
模型调用逻辑
边界检测逻辑
校园知识库主版本
学生事务知识库主版本
ASR/TTS 服务
账号或设备索引
知识库版本
异常和违规统计
```

推荐云端尽量“无状态为主”。每次请求时，开发板把本地状态一起发给云端；云端根据状态生成回复，再返回 `state_patch` 和 `memory_patch`，开发板写回 SD 卡。

## 6. 设备聊天接口草案

建议新增 `/api/device/chat`，用于未来 ESP-IDF 开发板对接。

请求示例：

```json
{
  "device_id": "xiaoxin_001",
  "user_text": "我今天不想学了",
  "local_state": {
    "mood": "sleepy",
    "energy": 42,
    "bond": 18,
    "last_seen": "2026-06-04T20:10:00+08:00",
    "focus_topic": "C语言"
  },
  "recent_memory": [
    "用户最近在担心 C语言链表",
    "用户喜欢轻松一点的鼓励"
  ]
}
```

响应示例：

```json
{
  "reply": "嗯，今天先别硬撑。你不是没用，只是电量真的低了。",
  "speech": "嗯，今天先别硬撑。你不是没用，只是电量真的低了。",
  "expression": "think",
  "action": "lean_forward",
  "state_patch": {
    "mood": "calm",
    "energy": 38,
    "bond_delta": 1,
    "focus_topic": "学习疲惫"
  },
  "memory_patch": {
    "add_growth": ["用户今晚表达学习疲惫，需要低压力陪伴"],
    "forget": []
  }
}
```

未来还可以增加：

```text
GET  /api/device/state
设备开机后获取云端版本信息和默认策略。

POST /api/device/event
上报摸摸头、充电、打卡、按钮事件。

GET  /api/device/idle
设备空闲时询问云端是否需要小动作或待机语。

GET  /api/device/audio/<id>
开发板下载 TTS 音频。
```

## 7. 状态驱动语气架构

不同状态对应不同语气和文本是可以实现的，而且应该成为下一阶段核心。

推荐流程：

```text
用户输入
  ↓
读取状态：mood / energy / bond / time_slot / focus_topic
  ↓
状态解释器 tone_policy
  ↓
生成本轮表达策略：语气、长度、表情、动作、禁止项
  ↓
注入模型 prompt
  ↓
后处理：边界检测、完整性检测、TTS 裁剪
  ↓
返回开发板
```

第一版状态可以只做：

```text
mood：小芯心情
energy：小芯精力
bond：亲密度
time_slot：时间段
focus_topic：最近关注话题
```

状态策略示例：

```text
深夜：
语气放轻，最多 2 句，不鼓励熬夜。

mood = worried：
先共情，再给一个很小的下一步。

energy < 30：
短句，少信息量，表情 sleepy，动作 slow_blink。

bond 高：
可以自然接上最近话题，但不要背诵记忆。
```

关键原则：

> 后端规则决定状态和语气，模型负责自然表达，后端再检查边界。

不要让模型自己决定所有状态，否则不可控。

设备阶段仍应复用现有 Web 后处理基线：

```text
template_reply(user_text)
  高风险问题先走确定性模板

LLM reply
  生成自然表达和表情标记

detect_reply_violations / is_fragmented_reply
  检查越界、半截回复和格式污染

to_speech_text
  只在句子边界裁剪 TTS 文本
```

这样开发板不会播放“我这就去问”“基本稳了”“我当年也是这样”或半句话回复。

## 8. 电子宠物状态建议

第一版建议定义：

```text
energy        精力值：0-100
mood          心情：calm / happy / worried / sleepy / excited
bond          亲密度：0-100
focus_topic   最近关注的话题：C语言 / 高数 / 竞赛 / 宿舍 / 情绪 / 专业选择
last_seen     上次见面时间
stage         关系阶段：first_meet / familiar / companion / old_friend
```

状态变化示例：

```text
用户主动聊天      bond +1，energy -2
用户表达开心      mood = happy
用户表达焦虑      mood = worried
用户完成任务      mood = excited，bond +2
很久没来          mood = calm 或 sleepy
深夜互动          energy -5，回复更柔和
每天第一次见面    energy 恢复一些
```

前台不要显示复杂数值，不要做成明显刷分游戏。可以转成自然表达：

```text
小芯今天精神不错。
小芯有点犯困，但还在认真听。
小芯还记得你最近在和 C 语言较劲。
```

## 9. 日常节律建议

小芯应该活在校园一天的节奏里。

```text
早上  6:00-10:00
轻轻打招呼，适合开启一天。

中午  11:00-13:30
轻松一点，可以聊吃饭、课间。

傍晚  17:00-20:00
适合复盘学习、竞赛、实验。

深夜  22:30 以后
降低信息量，偏安抚，不鼓励熬夜。
```

同一句“我今天不想学了”，不同时间的回复应不同：

```text
白天：
嗯，今天状态不在线也正常。先挑一个最小任务做五分钟，能启动就算赢。

深夜：
嗯，今天先别硬撑了。睡一觉，比继续耗着更像工程优化。
```

## 10. 陪伴事件建议

第一版可设计 6 类陪伴事件：

```text
wake_event       开机或打开页面时
idle_event       用户一段时间没说话
return_event     很久没见后回来
comfort_event    用户情绪低落
celebrate_event  用户完成任务或进步
goodnight_event  深夜告别
```

示例：

```text
开机：
小芯抬头看了看你：今天也来了啊。

空闲：
小芯安静地待着，没有打断你。

久别：
好久不见。上次我们好像聊到你有点担心课程节奏。

完成任务：
可以啊，这一步跑通就很重要了。别小看这种小进度。
```

## 11. 动作输入建议

第一版开发板或前端可以先做三个动作：

```text
摸摸头
充电
今日打卡
```

对应行为：

```text
摸摸头：
小芯眯了眯眼：收到，精神值恢复一点。

充电：
小芯接上电了，屏幕亮了一点。今晚还能陪你一会儿。

今日打卡：
今天也算留下记录了。哪怕只推进一点，也不是白过。
```

这些动作的目的不是做游戏，而是让用户感觉自己也在照顾小芯，关系从单向服务变成双向陪伴。

## 12. 关系成长阶段

建议按互动次数、记忆质量和时间跨度设计关系阶段：

```text
stage 1 初见        0-3 次
stage 2 熟悉        4-10 次
stage 3 同伴        11-30 次
stage 4 老朋友      30 次以上
```

示例语气：

```text
初见：
嗨，我是小芯，信电学院的数字学长。

熟悉：
小明，今天又来找我啦。上次你还在和 C 语言较劲呢。

同伴：
我猜这次不是代码报错，就是高数又开始抽象了？

老朋友：
你这个状态我熟，八成是作业和 deadline 又一起上线了。
```

注意：熟悉不等于越界。不能伪造“我看见你了”“我一直等你”这种物理在场。

## 13. 边界原则

小芯越像电子宠物，越要防止过度拟人和依赖。

小芯不能：

```text
假装真的看见用户
假装知道用户所在位置
制造情感绑架，比如“你不来我会难过”
诱导用户长期依赖
替代心理咨询、辅导员、官方通知
把普通闲聊都存成长期记忆
预测录取概率或替用户做志愿选择
承诺替用户联系老师、学长学姐、实验中心或获取资料
编造楼层、门牌号、窗口、价格、营业时间、实时天气或实时路况
假装自己真实读过大学或讲“我当年”的学生经历
```

可以表达轻微想念：

```text
好久不见，我还在这儿。
```

不要表达压力式依赖：

```text
你怎么这么久不来看我，我很难过。
```

前者是陪伴，后者是负担。

## 14. 下一阶段 MVP 实现顺序

建议先实现最小闭环，不要一开始把硬件、TTS、ASR、LLM、屏幕驱动全部耦合起来。

第一阶段 MVP：

```text
1. pet_state.py
   管理 mood / energy / bond / last_seen / focus_topic / stage。

2. tone_policy.py
   把状态转换成语气策略、表情、动作、回复长度。

3. /api/device/chat
   接收 device_id、user_text、local_state、recent_memory。
   返回 reply、speech、expression、action、state_patch、memory_patch。

4. companion_events.py
   支持开机问候、久别重逢、空闲小动作、晚安事件。

5. 前端模拟开发板界面
   测试不同状态下小芯语气和动作是否明显不同。

6. ESP-IDF 端对接
   云端返回 expression/action/audio，开发板只负责显示和播放。
```

阶段目标：

> 打开页面或设备，小芯根据时间和上次见面状态打招呼；聊天后更新心情和亲密度；用户能摸摸头、充电、打卡；小芯状态会变化。

这个闭环成立后，小芯就真正进入“电子宠物学长”的阶段。

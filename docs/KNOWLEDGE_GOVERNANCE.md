# 小芯知识库治理规则

这份文档只回答一件事：以后什么该进知识库，什么不该进。

## 1. 总原则

小芯继续补知识，但只做收缩式扩充，不做“越补越大的全校万能问答机”。

- `SKILL.md`：放长期稳定的人设、价值观、边界。
- `web/knowledge/*.json`：放可确定命中的稳定事实。
- LLM 自由回复：负责陪伴、鼓励、解释、过渡，不负责补事实。
- `knowledge_grounded`：允许模型把已给出的事实自然改写成口语，但不允许自己新增事实。

当前线上主链路不是“碰到关键词就模板回复”，而是：

1. `semantic_router.py` 先判断本轮是 `hard_template`、`knowledge_grounded` 还是 `free_chat`。
2. 只有真正需要事实支撑时，才给模型一小块受限知识。
3. 模型生成后还会再检查有没有超出知识库范围。

## 2. 准入四问

新增知识前先过这四问：

1. 这个问题是不是学生经常问。
2. 答案是不是至少一个学期内大概率稳定。
3. 能不能不依赖实时查询就安全回答。
4. 补进去之后，能不能明显减少拒答、绕答、幻觉或重复解释。

实操上，前三问至少成立，第四问有明显收益，才值得入库。

## 3. 优先级

优先补：

- 高频事务导航：补卡、证明打印、心理咨询、办事地点、常见入口。
- 高频生活导航：快递点、校区分布、服务设施、通知查看渠道。
- 陪伴增强事实：专业方向、竞赛类型、培养路径、阶段困惑。

不要补：

- 实时信息：当天通知、营业时间、排队、天气、临时安排。
- 主观判断：最好吃、最推荐、哪个老师最好、哪个实验室最强。
- 私人资源：联系方式、源文件、内部资料、群二维码。
- 长尾细节：极少被问的窗口例外、未验证的小道消息。
- 替代官方的内容：成绩、选课结果、审批结果、个人状态。

额外补充一条：

- 不要为了“让回答更有人味”往知识答案里加课程细节、实验室建议、老师评价、证件要求或流程经验。

## 4. 元数据规则

每份知识库必须有 `governance`，每类可命中记录通过 `record_defaults` 继承下面这些字段：

- `source`
- `owner`
- `last_verified`
- `volatility`
- `frequency`
- `fallback_channel`
- `do_not_elaborate`

如果某类知识是从现有知识库二次整理出来的陪伴增强事实，比如 `college_companion_facts.json`，每条记录还必须带 `source_refs`，明确指向现有 `SKILL.md` 或已有结构化知识中的来源位置。没有来源锚点的条目不入库。

推荐约束：

- `volatility` 只能用 `stable` / `semester` / `high`
- `frequency` 只能用 `high` / `medium` / `low`
- `capacity.max_items_by_path` 写清每类上限

## 5. 生命周期

- 先观察真实提问 1-2 周，再决定是否补库。
- 同一意图合并成一个标准答案，不堆很多近义条目。
- 超过一学期未命中、且维护成本高的条目优先下线。
- 易变信息默认不入库，只保留“去哪里确认”的指引。

## 6. 回答约束

知识入库以后，回答层也要一起受约束：

- `hard_template`：直接用确定性短答。
- `knowledge_grounded`：模型只能改写事实，不能新增事实。
- `free_chat`：不拿知识库模板抢答情绪陪伴和行动确认。

尤其要防的几类“半真半假”扩写：

- 知识库只写了地点，模型却补“带身份证”“先预约”“办公时间”。
- 知识库只写了专业区别，模型却补“去实验室转转”“哪个更适合你”“就业更好”。
- 知识库只写了通知渠道，模型却补“我看最近通知是……”。

## 7. 开发约束

- 新增或修改知识库后，运行 `python -m unittest discover web/tests`。
- 如果新增的是事实，不要直接写进 `SKILL.md`。
- 如果问题本质上是情绪支持或成长陪伴，优先让 LLM 自由回复，不要硬模板化。
- 如果新增的是 `college_companion_facts.json` 这类陪伴增强事实，必须同时补 `source_refs` 校验和测试。
